
# 002.md

##### Q:  

> How the ICN (Ibm Content Navigator) connected to the filenet/database repository
> * What the database save
> * What the filenet save

##### A:

**IBM Content Navigator (ICN) connects to FileNet through the Content Platform Engine (CPE), and FileNet itself stores metadata in a database and documents in a content storage area.**  
Below is a clear, structured explanation of how the pieces fit together and what each component stores, based on IBM’s official documentation.  

---

## 🧩 1. How IBM Content Navigator connects to FileNet  
ICN does **not** connect directly to the FileNet database.  
It connects to **FileNet Content Platform Engine (CPE)** using web services (CEWS/GraphQL/REST).  

When you configure a repository in ICN, you provide:  
- CPE URL  
- Object Store symbolic name  
- Authentication details  
  [IBM](https://www.ibm.com/docs/en/content-navigator/2.0.3?topic=navigator-connecting-configuring-filenet-content-manager-repositories)

ICN → CPE → Object Store → Database + Storage Area

---

## 🧩 2. How FileNet connects to the database repository  
FileNet CPE uses a **relational database** (DB2, Oracle, SQL Server) to store all metadata.  
This database is called the **GCD (Global Configuration Database)** and each Object Store has its own database schema.  
  [IBM Redbooks](https://www.redbooks.ibm.com/redbooks/pdfs/sg247547.pdf)

CPE reads/writes metadata through its internal object model and persists it in the database.

---

## 🧩 3. What the FileNet database stores  
The FileNet database stores **metadata only**, not the actual document files.

### The database contains:
- **Document metadata** (properties, title, creator, custom fields)  
- **Folder metadata**  
- **Class definitions**  
- **Property templates**  
- **Security permissions**  
- **Version history metadata**  
- **Workflow metadata**  
- **Object Store configuration**  
  [IBM Redbooks](https://www.redbooks.ibm.com/redbooks/pdfs/sg247547.pdf)

Think of the database as the “index” or “catalog” of the repository.

---

## 🧩 4. What FileNet stores outside the database  
The actual **document content** (PDFs, images, videos, etc.) is **not** stored in the database.

FileNet stores content in **Storage Areas**, which may be:
- File system storage  
- Network-attached storage  
- IBM Cloud Object Storage  
- Other pluggable storage systems  
  [IBM Redbooks](https://www.redbooks.ibm.com/redbooks/pdfs/sg247547.pdf)

Each document has:
- A metadata entry in the database  
- A pointer to the binary content in the storage area  

---

## 🧩 5. Putting it all together (architecture flow)

### **User → ICN → CPE → Database + Storage**
1. **User logs into ICN**  
   ICN authenticates and loads repository configuration.  
2. **ICN calls CPE**  
   Using CEWS/REST/GraphQL APIs.  
3. **CPE queries the database**  
   Retrieves metadata, folder structure, document properties.  
4. **If user opens a document**  
   CPE retrieves the content from the storage area.  
5. **ICN displays the document**  
   Using its built‑in viewer or external viewer.  

This architecture is confirmed in IBM’s installation and architecture documentation.  
  [publibfp.dhe.ibm.com](https://publibfp.dhe.ibm.com/epubs/pdf/c1937150.pdf)

---

## 🧩 6. Summary Table

| Component | Stores | Does NOT Store |
|----------|--------|----------------|
| **ICN** | UI configuration, desktops, plugins | No FileNet content |
| **CPE (FileNet)** | Manages metadata, security, API access | No binary content |
| **FileNet Database (GCD + Object Store DB)** | Metadata, classes, properties, security, workflow | No document files |
| **FileNet Storage Areas** | Binary content (PDF, images, videos) | No metadata |
