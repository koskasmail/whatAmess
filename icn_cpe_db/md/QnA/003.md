## Authentication between IBM Content Navigator and FileNet

### 1. High-level flow

**User → ICN (web app) → Application server security → CPE (FileNet)**

1. **User logs into ICN**
   - **ICN** is a web application (WebSphere, WebLogic, Liberty, etc.).
   - Authentication is handled by the **application server’s security realm** (LDAP/AD, SSO, etc.).

2. **ICN obtains a subject/credential**
   - After login, the app server creates a **JAAS Subject** (or equivalent) representing the user.
   - ICN uses this subject to call **Content Platform Engine (CPE)**.

3. **ICN connects to CPE**
   - ICN uses **CEWS/REST** calls to CPE.
   - The user identity is propagated via:
     - **LTPA / SPNEGO / SSO token**, or  
     - **Username/password** (for basic setups), or  
     - **OAuth/OpenID Connect** in more modern deployments.

4. **CPE validates the identity**
   - CPE is also deployed on an application server with its own **security configuration** pointing to the same or a trusted LDAP/AD.
   - The token/credentials are validated; CPE maps the identity to a **FileNet user**.

5. **Authorization**
   - Once authenticated, CPE checks **ACLs and security** on:
     - Object Stores  
     - Folders  
     - Documents  
     - Classes and properties  
   - ICN only shows what the user is allowed to see.

---

### 2. Common authentication patterns

- **LDAP/Active Directory + LTPA**
  - **Most common** in enterprise.
  - Both ICN and CPE app servers share the same **LTPA keys**.
  - Single sign-on across ICN, CPE, and possibly other apps.

- **Kerberos / SPNEGO**
  - For **true SSO** from Windows desktop (no login prompt).
  - Browser gets a Kerberos ticket, sends it to ICN, which passes identity to CPE.

- **SAML / OIDC / OAuth**
  - ICN and CPE trust an **Identity Provider (IdP)**.
  - User authenticates to IdP; ICN/CPE receive tokens and map them to LDAP users.

- **Basic username/password**
  - Often used in **dev/test**.
  - ICN stores repository credentials (or prompts user) and passes them to CPE.

---

### 3. Key design tips for authentication

- **Use a single identity source**  
  Same LDAP/AD (or trusted domains) for ICN and CPE.

- **Align group-based security**  
  Use **AD groups** for FileNet permissions instead of individual users.

- **Plan for SSO early**  
  Changing from basic auth to SSO later can be painful.

- **Secure transport**  
  Always use **HTTPS** between:
  - Browser ↔ ICN  
  - ICN ↔ CPE

## Designing Object Stores

Think of an **Object Store** as a logical repository: schema + security + metadata + link to storage.

### 1. How many Object Stores?

**Typical patterns:**

- **Single Object Store**
  - **Pros:** Simple, easy to manage, shared classes and properties.
  - **Cons:** Harder isolation between departments/tenants.

- **Multiple Object Stores**
  - **Pros:** Strong isolation (security, retention, backup, lifecycle).
  - **Cons:** More admin overhead, more complexity in ICN configuration.

**Common design choices:**

- **Per environment:** DEV, TEST, PROD each have their own domain/object stores.
- **Per business domain:** e.g., HR, Finance, Claims, Legal.
- **Per tenant:** in multi-tenant scenarios.

---

### 2. What goes into an Object Store?

- **Classes**
  - **Document classes** (e.g., Invoice, Contract, PolicyDocument).
  - **Folder classes** (e.g., CustomerFolder, CaseFolder).
  - **Custom objects** (for configuration, references, etc.).

- **Property templates**
  - Reusable metadata fields (CustomerID, InvoiceNumber, CaseID).
  - Assigned to classes.

- **Security model**
  - Default ACLs on classes.
  - Role-based access using groups.

- **Retention and lifecycle**
  - Disposition policies (e.g., delete after X years).
  - Holds and legal retention.

---

### 3. Object Store design best practices

- **Model business entities, not just “documents”**
  - Create specific document classes with meaningful properties.
- **Reuse property templates**
  - Avoid duplicate definitions of the same concept (e.g., CustomerID).
- **Keep security manageable**
  - Use **group-based ACLs** and class-level defaults.
- **Plan for growth**
  - Consider future classes and integrations when designing the schema.

## Designing storage areas

Storage areas define **where the binary content lives**.

### 1. Types of storage areas

- **File Storage Area (FSA)**
  - Content stored on a **file system** (local or NAS).
  - Simple, common, good for many workloads.

- **Fixed Content Device / Object Storage**
  - IBM Cloud Object Storage, EMC Centera, S3-compatible, etc.
  - Good for **long-term archiving**, WORM, compliance.

- **Advanced storage**
  - Tiered storage, encryption, compression, etc., depending on platform.

---

### 2. How storage areas relate to Object Stores

- Each **Object Store** can have **one or more storage areas**.
- A document class can be configured to use:
  - A **default storage area**, or  
  - A **specific storage area** (e.g., large videos vs. small PDFs).

When a document is created:
- Metadata goes to the **Object Store database**.
- Content is written to the **configured storage area**.
- The database stores a **pointer** to the content location.

---

### 3. Storage area design strategies

- **By content type**
  - **Example:**  
    - PDFs, office docs → Standard FSA  
    - Images/scans → High-capacity NAS  
    - Long-term archives → Object Storage / WORM

- **By performance**
  - Frequently accessed content → Faster storage.  
  - Old/rarely accessed content → Cheaper, slower storage.

- **By retention/compliance**
  - Regulated documents → WORM-capable storage area.  
  - Non-critical docs → Standard storage.

- **By size**
  - Very large files (videos, CAD) in dedicated storage areas to manage backup and performance.

---

### 4. Storage area best practices

- **Separate content from database**
  - Never store large binaries in DB; FileNet already separates them.

- **Plan backup/restore**
  - Database backup must be **consistent** with storage area backup.
  - Document pointers in DB must match actual files.

- **Monitor capacity and performance**
  - Watch growth trends.
  - Use multiple storage areas if a single one becomes a bottleneck.

- **Security at storage level**
  - Even though FileNet controls access, ensure **file system/object storage** is also secured (OS permissions, encryption).

---

## Quick mental picture

- **ICN**  
  UI + configuration; authenticates via app server; calls CPE.

- **CPE / FileNet**
  Enforces security, manages metadata, talks to DB and storage.

- **Database**
  Stores **metadata, classes, properties, ACLs, workflows**.

- **Storage Areas**
  Store **binary content**; DB holds pointers to them.
